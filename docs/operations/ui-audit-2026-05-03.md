# Career Copilot full UI audit

_Date: 2026-05-03_

## Scope and method

This is a code-level UI audit across App Router pages and shared UI components, with emphasis on:

- Visual consistency and design-system reuse
- Navigation and information architecture coherence
- Accessibility and interaction quality
- Responsiveness and layout behavior
- Governance-sensitive UI surfaces (admin, eligibility, notifications)

Primary inventory inputs:

- `app/` routes and layout files
- `components/` UI/component tree
- global style foundation and shared UI primitives

## Executive summary

UI maturity is **medium-high** with strong momentum from Sprint 8 light-theme migration, but the codebase currently runs a **hybrid styling architecture** (legacy utility classes + new `cc-*` design primitives + inline styles). This causes consistency and maintainability risk despite good feature coverage.

Top priorities:

1. Finish light-theme rollout by deprecating legacy hard-coded color classes and inline style blocks in shared shells.
2. Consolidate primitive usage so feature pages consistently consume `components/ui` rather than bespoke card/button/tag patterns.
3. Close accessibility gaps (focus visibility, semantic landmarks, contrast checks, keyboard order in dense dashboard/admin surfaces).
4. Reduce navigation duplication between root and dashboard/admin shells.

## Route and surface coverage audit

### 1) Layouts and shell patterns

Observed shells:

- Root layout: sticky top nav and top-level route tabs (`/today`, `/exams`, `/study`, `/community`, `/marketplace`, `/profile`).
- Dashboard layout + dashboard nav shell for authenticated experience.
- Admin layout for governance/control surfaces.

Findings:

- Root and dashboard both present navigation paradigms; this is useful for context separation but risks duplication of labels, active-state behavior, and visual grammar.
- Root layout still relies heavily on inline style objects, which weakens tokenized theming portability.

### 2) Dashboard experience

Strengths:

- Dashboard shell composition is modular and scalable (`TodayPrioritiesPanel`, `NextBestActionPanel`, mission control, notifications, study widgets, AI widget).
- Deterministic workflows (priorities, actions, mission control) are surfaced in actionable cards instead of passive stats-only surfaces.
- Right-rail information density is good for aspirant workflows.

Risks:

- Mixed styling systems in adjacent widgets may fragment hierarchy and spacing rhythm.
- Dense card stacks need stronger keyboard/focus traversal cues and heading-level consistency.

### 3) Exam/recruitment and study surfaces

Strengths:

- Exam/recruitment and study flows are represented in dedicated pages and components (`/dashboard/exams`, recruitment detail, study-plan variants).
- Timeline/status components support progress narrative and state clarity.

Risks:

- Some older pages likely still carry pre-light-theme visual language and should be normalized using current tokens.
- CTA styles and tag/badge semantics vary across modules, reducing pattern predictability.

### 4) Community and marketplace surfaces

Strengths:

- Clear module separation (`components/forum/*`, `components/marketplace/*`) with room for independent iteration.
- Existing community visual conventions (channel tabs, flair badges, quota/lock banners) provide a workable design base.

Risks:

- Community and marketplace patterns appear partly bespoke; alignment with shared primitive layer is incomplete.
- Moderation and trust-state affordances in future admin/community flows should be standardized now to avoid style drift.

### 5) Admin and governance surfaces

Strengths:

- Extensive admin route coverage exists (RBAC, audit, eligibility queue, notifications, scrape/sources, control-support).
- Governance tooling is visible and operationally central, matching product principles.

Risks:

- Admin UI pattern library appears fragmented (`components/admin/ui/index.tsx` plus local forms/panels), increasing long-term UX inconsistency.
- High-density admin tables/forms require stricter accessibility and interaction standards than currently documented.

## Design-system and styling audit

### Current state

The codebase uses three concurrent styling approaches:

1. Legacy global classes in `app/globals.css` (`.card`, `.btn`, `.tag-*`, etc.)
2. Shared primitive layer in `components/ui/index.tsx` (`Card`, `Pill`, `Button`, etc.)
3. Inline style objects (notably in root layout and some legacy pages)

### Impact

- Slower UI change velocity due to duplicated style logic.
- Higher risk of inconsistent visual updates during light-theme completion.
- Harder accessibility verification because interaction states are spread across systems.

### Recommendation

Adopt a strict migration sequence:

- Phase A: freeze new legacy class usage and inline style introduction.
- Phase B: map all reusable patterns to `components/ui` primitives.
- Phase C: retire superseded legacy classes once coverage reaches >90% of active pages.

## Accessibility and interaction audit

### Positive signals

- Semantic page structures and card segmentation are broadly present.
- Notification and priorities modules suggest task-oriented information hierarchy.

### Gaps to validate and harden

- Ensure visible focus states for all interactive elements, especially nav links/buttons rendered via inline styles.
- Verify heading hierarchy (`h1`→`h2` flow) on dashboard/admin pages with many cards.
- Add/confirm ARIA labels for icon-only controls and compact admin actions.
- Check contrast ratios for muted text/tags in light theme.
- Validate keyboard-only access for dense dashboards, feeds, and admin queues.

## Responsiveness audit

### Strengths

- Global helpers include mobile grid fallback.
- Dashboard shell uses responsive grid patterns with single-column fallback.

### Risks

- Widget-level breakpoints are uneven when components use mixed style systems.
- Horizontal-overflow components (tabs/lifecycles) need consistent touch-target and scroll affordances.

## Information architecture audit

### Strengths

- User mental model is clear: Today → Exams → Study → Community → Marketplace → Profile.
- Dashboard surfaces deterministic next steps and profile blockers effectively.

### Risks

- Potential route overlap between top-level routes and dashboard equivalents can confuse users if visual states differ.
- Naming parity should be audited to ensure identical terms for equivalent concepts across root/dashboard/admin contexts.

## Prioritized action plan

### P0 (release trust + UX stability)

1. Standardize focus-visible and keyboard states across nav, cards, and button primitives.
2. Remove or encapsulate inline styling from root shell into tokenized classes/primitives.
3. Normalize card/button/tag variants used in mission-control and admin workflows.

### P1 (next sprint)

1. Complete light-theme parity audit page-by-page (dashboard, exams, study, community, marketplace, profile, admin).
2. Publish a UI pattern map (component → approved variant → usage examples).
3. Add automated accessibility checks (axe/lint rules) in CI for critical routes.

### P2 (strategic follow-up)

1. Introduce visual regression snapshots for critical aspirant/admin surfaces.
2. Create governance-state design tokens (verified/pending/rejected/escalated) shared across scraper, eligibility, and moderation UIs.
3. Build an admin interaction standard for queues, filters, and bulk actions.
